ProjectoLad Firestore Data Model
================================

Primary collection
- Collection: events
- One document per live or scheduled event

Why the model was expanded
- The old demo model was mostly display text.
- A real booking app needs structured fields so the UI can sort, filter, format dates, show live inventory, and evolve without rewriting strings by hand.

Recommended event document fields
- title: string
- subtitle: string
- summary: string
- heroSymbol: string
- badgeText: string
- hostName: string
- venueName: string
- neighborhood: string
- address: string
- startDate: timestamp
- endDate: timestamp
- doorsOpen: timestamp or null
- priceFrom: number
- currencyCode: string
- description: string
- agePolicy: string
- dressCode: string
- parkingInfo: string
- musicGenres: array<string>
- lineup: array<string>
- amenities: array<string>
- status: string
  Allowed values:
  - scheduled
  - soldOut
  - cancelled
- isFeatured: boolean
- capacity: number or null
- ticketsRemaining: number or null
- location: map
  - latitude: number
  - longitude: number
- ticketTiers: array<map>
  Each tier should contain:
  - code: string
  - name: string
  - price: number
  - currencyCode: string
  - perks: array<string>
  - remainingInventory: number or null
  - isRecommended: boolean

Good additional details to add later if the product grows
- coverImageURL for remote event artwork
- organizerContact for support or last-minute changes
- refundPolicy for checkout transparency
- arrivalWindowStart and arrivalWindowEnd for timed entry
- tableMinimum for hospitality-driven venues
- socialLinks for artist or venue promotion
- waitlistEnabled for sold-out events
- walletPassTemplateID if Apple Wallet issuance is added

Indexing notes
- Start with an ascending index on startDate
- If you filter featured events often, add a composite index on isFeatured + startDate
- If you later query by status + date range, add a composite index on status + startDate

Current app behavior
- The app listens to the events collection in Firestore.
- Documents are decoded directly into the Event model.
- If Firestore is empty or unavailable, the app falls back to local sample events so the UI still works.
